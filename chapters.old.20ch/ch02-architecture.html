2.2 LLM：Agent 的大脑

LLM 是 Agent 的核心推理引擎。但 Agent 中的 LLM 和普通 ChatBot 中的 LLM 用法不同：

ChatBot 中的 LLM
用户输入 → LLM → 直接输出
单轮调用，输入输出
不需要结构化输出
不需要理解工具描述
角色：对话引擎
Agent 中的 LLM
输入 → LLM → 结构化决策
多轮调用，每轮做一次决策
需输出 JSON 格式的行动计划
需理解工具描述并选择正确工具
角色：推理引擎 + 决策中心

System Prompt：Agent 的"岗位说明书"

Agent 中 LLM 的行为由 System Prompt 控制。一个好的 System Prompt 就像一份岗位说明书：

System Prompt 示例 
你是一个数据分析 Agent。

你的能力：
- 可以调用 search_web 工具搜索信息
- 可以调用 run_python 工具执行代码
- 可以调用 write_file 工具写入文件

你的工作流程：
1. 分析用户需求，判断需要哪些工具
2. 制定执行计划，逐步调用工具
3. 观察每次工具返回的结果
4. 如果结果不完整，调整计划继续
5. 收集足够信息后，整理最终回答

输出格式：
当需要调用工具时，输出 JSON：
{"action": "工具名", "input": {"参数": "值"}}

当有最终答案时，输出：
{"answer": "你的回答"}

2.3 记忆系统：Agent 的时间维度

记忆让 Agent 拥有"过去"。没有记忆的 Agent 就像每天失忆的人，每次对话都从零开始。

🧠 记忆的三层结构
短期记忆

= LLM 上下文窗口
存当前对话历史
容量有限（4K~200K token）
速度最快

长期记忆

= 向量数据库
存历史交互经验
容量无限
需语义检索

工作记忆

= 刮擦板/便签纸
存当前任务的中间状态
任务结束即清除
辅助多步推理

短期记忆的压缩问题

LLM 的上下文窗口有限。对话越长，早期内容会被"挤出"窗口。Agent 需要管理这个问题：

长期记忆的检索过程

长期记忆存在向量数据库中（如 Chroma、Pinecone）。检索过程：

长期记忆检索流程 
1. 用户输入 → Embedding 模型 → 向量 [0.12, 0.85, ...]
2. 在向量数据库中做相似度搜索 (top-k=5)
3. 取回最相关的 5 条历史记忆
4. 将记忆注入 prompt → "你之前处理过类似问题..."
5. LLM 基于记忆 + 当前输入生成回答

2.4 规划系统：Agent 的思维深度

规划是 Agent 区别于简单工具调用的关键。面对复杂任务，Agent 不是"想一步做一步"，而是先制定完整计划。

任务拆解：把大问题变成小问题

📋 任务拆解示例

用户请求："帮我调研 LangGraph 框架，写一份技术报告"

Agent 规划：
① 搜索 LangGraph 官方文档和教程 → search_web
② 搜索 LangGraph GitHub 仓库和 Star 数 → search_web
③ 搜索 LangGraph vs 其他框架对比 → search_web
④ 整理信息，生成报告大纲 → llm_reasoning
⑤ 逐节撰写报告内容 → llm_reasoning
⑥ 将报告保存为文件 → write_file

三种规划策略

策略 原理 适用场景
CoT（思维链） Let's think step by step 简单推理任务
ToT（思维树） 分支探索多个方案，选最优 需要搜索的问题
ReAct 思考→行动→观察循环 需要工具的交互任务

2.5 工具系统：Agent 的手脚

工具让 Agent 能与外部世界交互。常见的工具类型：

🔍 信息获取类

搜索引擎、天气API、新闻API、数据库查询

⚙️ 代码执行类

Python REPL、Shell命令、沙箱执行

📝 文件操作类

读写文件、创建目录、搜索文件

🌐 通信类

发邮件、调API、发消息、 webhook

MCP：工具的 USB 接口

2024 年 Anthropic 推出的 MCP（Model Context Protocol） 正在成为工具调用的标准协议。可以把 MCP 理解为"工具的 USB 接口"：

以前每个框架自己定义工具格式（像各品牌手机充电器），MCP 统一了接口（像 USB-C），一个 MCP Server 写好后，所有支持 MCP 的 Agent 框架都能用。

2.6 四模块如何协同工作

把四个模块串起来，看一个完整的执行流程：

四模块协同伪代码 
# 用户输入
user_input = "帮我查一下最近的 AI Agent 论文并总结"

# ① 感知：LLM 理解意图
intent = llm.analyze(user_input)
# → intent: "搜索学术论文 + 总结摘要"

# ② 记忆：检索相关历史
context = memory.retrieve(user_input, top_k=3)
# → context: ["用户之前关注LangGraph", "用户偏好简洁总结"]

# ③ 规划：拆解任务
plan = planner.decompose(intent, context)
# → plan: ["搜索arxiv", "下载top5论文", "逐篇摘要", "综合总结"]

# ④ 执行：逐步调用工具
for step in plan:
    if step.needs_tool:
        result = tools[step.tool_name](step.tool_input)
    else:
        result = llm.reason(step.prompt, context)
    
    # 观察结果，更新记忆
    memory.add(result)
    context = memory.retrieve(user_input, top_k=5)
    
    # 判断是否需要调整计划
    if not step.is_complete(result):
        plan = planner.adjust(step, result, plan)

# 返回最终结果
final_answer = llm.generate(user_input, context)
print(final_answer)

2.7 八种 Agent 设计模式详解

前面我们了解了 Agent 的四大模块。但模块如何组合、流程如何编排，决定了 Agent 的"性格"和"能力边界"。业界已总结出八种经典设计模式，每种模式解决一类核心问题。

选择设计模式就像选择武功流派——没有"最强的模式"，只有"最适合当前任务的模式"。理解每种模式的原理和适用边界，才能设计出优秀的 Agent。

2.7.1 ReAct 模式：思考→行动→观察循环

ReAct（Reasoning + Acting）是最经典的 Agent 设计模式，在 2.4 节已有简要介绍。这里我们深入其内部机制。

🔄 ReAct 核心循环

每一轮执行三个步骤，循环往复直到任务完成：

Thought（思考）

LLM 分析当前状态
决定下一步做什么
输出推理过程

Action（行动）

调用外部工具
获取新信息
改变外部状态

Observation（观察）

接收工具返回结果
更新认知状态
决定是否继续

适用场景：需要与外部世界交互的动态任务——信息检索、多步推理、实时数据查询。

✅ 优点

推理过程可追溯、可调试
每步都有观察反馈，容错性强
自然处理不确定性，动态调整路径

⚠️ 缺点

循环次数多，token消耗大
可能陷入死循环（需设置最大轮次）
每步单独推理，缺乏全局视野

ReAct 模式伪代码 
# ReAct 循环
max_turns = 10
history = []

for turn in range(max_turns):
    # Thought：LLM 分析当前状态并推理
    thought = llm.generate(
        prompt=f"当前状态：{history}\n下一步应该做什么？",
        system="你是ReAct Agent，先思考再行动"
    )
    
    # Action：解析并执行工具调用
    action = parse_action(thought)  # → {tool: "search", input: "LangGraph"}
    observation = execute_tool(action)  # → "LangGraph 是一个..."
    
    # Observation：记录结果
    history.append({"thought": thought, "action": action, "observation": observation})
    
    # 判断是否完成
    if is_final_answer(thought):
        return thought.final_answer

return "未能完成任务（达到最大轮次）"

2.7.2 Plan-and-Execute 模式：先规划后执行

Plan-and-Execute 将 Agent 的决策拆分为两个阶段：规划器一次性制定完整计划，执行器逐步执行，重规划器在遇到偏差时修正。

适用场景：复杂多步骤任务——项目规划、数据分析流水线、长流程自动化。

✅ 优点

全局视角，计划更连贯
执行效率高，减少无效循环
计划可审核、可人工干预

⚠️ 缺点

初始计划可能不完整
重规划成本高（需要重新推理）
对简单任务过度设计

Plan-and-Execute 模式伪代码 
# Plan-and-Execute
plan = planner.generate(user_input)  # 一次性制定完整计划
# → [步骤1: 搜索, 步骤2: 分析, 步骤3: 写报告]

for step in plan:
    result = executor.run(step)  # 执行单步
    
    # 重规划检查
    if not meets_expectation(result, step.expected_output):
        plan = replanner.adjust(
            original_plan=plan,
            failed_step=step,
            actual_result=result
        )
        # → 从失败步骤开始重新规划后续步骤

return aggregate_results(plan)

2.7.3 Multi-Agent 模式：多 Agent 协作

Multi-Agent 模式将复杂任务分发给多个专职 Agent，每个 Agent 负责一个领域，通过层级委派或平等协作完成任务。

👥 Multi-Agent 两种组织形式
层级委派（Orchestrator）

主 Agent 像项目经理
拆解任务→分配给子Agent
汇总结果→输出最终答案

平等协作（Swarm）

无主Agent，各自负责领域
通过握手协议交接任务
适合流水线式工作流

适用场景：跨领域复杂任务——软件开发（需求+设计+编码+测试）、多角色协作（调研+撰写+审核）。

✅ 优点

每个Agent专精一个领域
并行执行，效率高
可扩展，新增Agent即可覆盖新领域

⚠️ 缺点

协调复杂，信息传递易丢失
Agent间共识困难（理解偏差）
资源开销大（多个LLM调用）

Multi-Agent 层级委派伪代码 
# Multi-Agent (Orchestrator 模式)
orchestrator = Agent("项目经理", tools=["delegate"])
researcher = Agent("调研员", tools=["search_web"])
writer = Agent("撰写员", tools=["write_file"])
reviewer = Agent("审核员", tools=["read_file"])

# 主Agent拆解并分配任务
plan = orchestrator.plan("写一份AI Agent调研报告")
# → 子任务1: 调研, 子任务2: 撰写, 子任务3: 审核

# 执行流水线
research = researcher.execute(plan.step1)  # → 调研结果
draft = writer.execute(plan.step2, context=research)  # → 初稿
final = reviewer.execute(plan.step3, input=draft)  # → 审核意见

# 主Agent汇总
result = orchestrator.aggregate(research, draft, final)

2.7.4 Reflective 模式：反思校验

Reflective 模式让 Agent 在执行后自我审视，评估结果质量，发现问题后修正重做。代表技术：Reflexion 和 Self-RAG。

适用场景：高质量输出任务——代码生成（需自测）、论文写作（需自审）、数据分析（需自验）。

✅ 优点

输出质量显著提升
自我纠错减少人工干预
反思记录可沉淀为经验

⚠️ 缺点

额外token消耗（反思轮次）
反思标准难定义
可能过度反思（无限循环修正）

Reflective 模式伪代码 
# Reflexion 模式
max_reflections = 3

for attempt in range(max_reflections):
    # 执行任务
    result = agent.execute(task)
    
    # 反思：评估结果
    critique = llm.evaluate(
        prompt=f"评估以下结果的质量和缺陷：\n{result}\n\n指出问题并给出改进建议"
    )
    
    # 判断是否通过
    if critique.quality_score >= threshold:
        return result  # 质量达标，输出
    
    # 未通过：将反思注入下次尝试的上下文
    agent.memory.add(
        f"上次尝试的问题：{critique.issues}\n改进方向：{critique.suggestions}"
    )

return result  # 达到最大反思次数，返回最后一次结果

2.7.5 Tool-Augmented 模式：工具门禁

Tool-Augmented 模式的核心思想：只在需要时才调用工具。LLM 先自行推理，判断是否需要外部帮助，需要时才"开门"调用工具。

传统 Agent
每步都尝试调用工具
工具调用是默认行为
大量无效调用
token浪费严重
Tool-Augmented
先推理判断是否需要工具
工具调用是"门禁"式
只在必要时才调用
高效节省 token

适用场景：混合型任务——大部分推理可内部完成，只在关键节点需要外部数据。

✅ 优点

减少无效工具调用
降低延迟和成本
保留LLM自身推理能力

⚠️ 缺点

判断"是否需要工具"本身可能出错
过于依赖LLM的自省能力
复杂任务中门禁判断成本高

Tool-Augmented 模式伪代码 
# Tool-Augmented (门控调用)
response = llm.generate(user_input)

# 解析LLM输出，判断是否需要工具
if contains_tool_call(response):
    # 需要工具 → 开门
    tool_result = execute_tool(response.tool_call)
    final = llm.generate(
        prompt=f"基于工具结果完善回答：{tool_result}\n原问题：{user_input}"
    )
else:
    # 不需要工具 → 直接返回LLM回答
    final = response

return final

2.7.6 Memory-Augmented 模式：记忆治理

Memory-Augmented 模式不是简单的"加个向量数据库"，而是系统性地管理短期、长期、工作记忆的协同。核心是让 Agent 在正确的时刻获取正确的记忆。

🧠 三层记忆协同机制

三层记忆不是独立的，而是通过"治理策略"协同工作：

写入策略：对话内容 → 短期记忆；关键决策 → 长期记忆；中间推理 → 工作记忆

检索策略：当前输入 → 短期记忆（近邻）；语义相关 → 长期记忆（相似度）；任务状态 → 工作记忆（精确匹配）

淘汰策略：短期记忆超限 → 摘要压缩后存长期；工作记忆任务结束 → 清除；长期记忆定期 → 按重要性衰减

适用场景：长期运行 Agent——客服系统（记住用户偏好）、个人助手（积累用户习惯）、研究助手（追踪项目进展）。

✅ 优点

Agent 越用越"聪明"
上下文精准，减少注意力稀释
支持跨会话连续性

⚠️ 缺点

记忆检索增加延迟
存储和索引成本高
记忆可能过期或冲突

Memory-Augmented 模式伪代码 
# Memory-Augmented (三层记忆协同)
def handle_input(user_input):
    # 检索三层记忆
    recent = short_term_memory.get_recent(k=5)      # 短期：最近5轮对话
    related = long_term_memory.search(user_input, k=3) # 长期：语义相关3条
    state = working_memory.get_current_state()        # 工作：当前任务状态
    
    # 组装上下文
    context = assemble(recent, related, state)
    
    # LLM 推理
    response = llm.generate(user_input, context=context)
    
    # 记忆写入
    short_term_memory.add(user_input, response)           # 写入短期
    if is_important(response):                             # 判断重要性
        long_term_memory.store(response, metadata={...})   # 关键信息写入长期
    working_memory.update(response.intermediate_state)     # 更新工作记忆
    
    # 记忆淘汰
    if short_term_memory.size > threshold:
        summary = llm.summarize(short_term_memory.oldest)
        long_term_memory.store(summary)
        short_term_memory.prune_oldest()
    
    return response

2.7.7 RAG 模式：检索增强生成

RAG（Retrieval-Augmented Generation）将外部知识库检索与 LLM 生成结合。Agent 先从知识库检索相关文档，再基于检索结果生成回答，而非纯靠 LLM 的参数记忆。

适用场景：知识密集型任务——企业知识问答、技术文档查询、法律条文检索、医学文献分析。

✅ 优点

回答基于事实，减少幻觉
知识库可实时更新
无需重新训练模型

⚠️ 缺点

检索质量决定回答质量
知识库构建和维护成本
检索延迟影响响应速度

RAG 模式伪代码 
# RAG (检索增强生成)
def rag_answer(query):
    # Step 1: 向量化用户问题
    query_embedding = embed_model.encode(query)
    
    # Step 2: 在知识库中检索相关文档
    retrieved_docs = vector_db.search(
        query_embedding, top_k=5, threshold=0.7
    )
    # → [doc1: "LangGraph 支持循环状态图", doc2: "...", ...]
    
    # Step 3: 组装增强 Prompt
    augmented_prompt = f"""
    基于以下参考资料回答问题：
    {format_docs(retrieved_docs)}
    
    问题：{query}
    
    如果参考资料不足以回答，请明确说明。
    """
    
    # Step 4: LLM 生成回答
    answer = llm.generate(augmented_prompt)
    return answer, retrieved_docs  # 返回答案+来源

2.7.8 Autonomous Loop 模式：自主循环

Autonomous Loop 让 Agent 目标驱动、持续执行，不需要人类每步指令。Agent 接收一个高层目标后，自主规划、执行、评估、调整，直到达成目标或耗尽资源。

🎯 Autonomous Loop 的核心特征
目标驱动：给定高层目标（如"优化这个数据库"），Agent 自行拆解和执行

持续执行：不等待人类确认每一步，自动推进

自我评估：每步完成后自检，判断是否达成子目标

动态调整：遇到阻碍时自动切换策略

终止条件：目标达成 / 达到最大步数 / 资源耗尽 / 人工中断

适用场景：长周期自动化任务——代码重构（自主扫描+修改+测试）、系统运维（自主巡检+修复）、数据清洗（自主检测+处理+验证）。

✅ 优点

高度自动化，最小人工干预
可处理长周期复杂任务
具备自我纠错能力

⚠️ 缺点

失控风险（可能偏离目标）
资源消耗不可预测
需要严格的安全边界

Autonomous Loop 模式伪代码 
# Autonomous Loop (自主循环)
def autonomous_run(goal, max_steps=50, budget=1000):
    state = {"goal": goal, "progress": [], "steps_used": 0}
    
    while state.steps_used < max_steps and budget > 0:
        # 自主规划下一步
        next_step = llm.plan(
            goal=state.goal,
            progress=state.progress,
            remaining_budget=budget
        )
        
        # 执行
        result = execute(next_step)
        state.steps_used += 1
        budget -= calculate_cost(result)
        
        # 自我评估
        evaluation = llm.evaluate(
            "这个结果是否推进了目标？", result, state.goal
        )
        
        if evaluation.goal_achieved:
            return result  # 目标达成！
        
        # 记录进展，动态调整
        state.progress.append({"step": next_step, "result": result, "eval": evaluation})
        
        if evaluation.should_change_strategy:
            state.goal = evaluation.refined_goal  # 调整子目标
    
    return "未能达成目标"  # 资源耗尽

2.8 Context Engineering（上下文工程）

如果说 Prompt Engineering 是教 Agent "怎么说话"，那么 Context Engineering 是教 Agent "在什么环境中说话"。这是 2025 年 Agent 开发领域最重要的认知升级。

从 Prompt Engineering 到 Context Engineering

Prompt Engineering
关注：怎么写指令
范围：单条 Prompt 文本
假设：LLM 只需要好指令
局限：忽略了LLM的"环境"
Context Engineering
关注：怎么组装完整信息环境
范围：Prompt + 历史 + 工具 + 知识
认知：LLM 需要"正确的上下文"
优势：系统性管理信息输入
Prompt Engineering 问的是"你怎么告诉LLM做事"。Context Engineering 问的是"你给LLM的工作环境里有什么"。就像一位厨师——好的食谱（Prompt）固然重要，但新鲜的食材、合适的厨具、安静的环境（Context）才是做出好菜的前提。

Context 的三要素

📦 Context 的完整组成
用户输入

当前请求
问题/指令/目标
即时意图

补充资料

检索到的文档
工具返回结果
知识库片段

历史问答

之前对话轮次
已执行的操作记录
积累的推理痕迹

Context Rot：注意力稀释问题

随着对话增长，Context 中充斥着大量低价值信息，LLM 的注意力被稀释，关键信息被淹没——这就是 Context Rot（上下文腐化）。

🧪 Context Rot 的表现
症状 1：LLM "忘了"之前确认的关键约束（如"不要用第三方库"）

症状 2：回答引用了过时信息（早期对话中已被修正的错误）

症状 3：对简单问题给出复杂但不准确的回答

根因：Context 太长 → LLM 注意力分布均匀 → 关键信息权重被稀释

Just-in-Time Context：即时上下文

解决 Context Rot 的核心理念：Just-in-Time Context——只在需要的时刻注入需要的信息，而非一次性塞满。

一次性塞满
启动时加载全部历史
所有工具描述一次性注入
知识库全量检索
→ Context Rot
Just-in-Time
每步按需检索相关记忆
只注入当前步骤需要的工具
动态检索最相关的片段
→ 精准高效

上下文压缩策略

当 Context 不可避免地增长时，需要压缩策略保留核心信息：

策略 原理 适用场景 风险
截断 直接丢弃最旧的消息 实时性强的对话 可能丢失关键约束
摘要 LLM 对旧内容生成摘要替代原文 通用场景 摘要可能遗漏细节
滑动窗口 只保留最近N轮对话 短期对话型Agent 丢失长期上下文
选择性保留 按重要性评分保留关键消息 决策密集型任务 评分标准难定义
语义保护型压缩 压缩低语义密度内容，保护关键语义 高精度Agent 实现复杂度高

[User] 现在帮我改成降序
[Agent] # 只保留关键上下文，LLM注意力集中
def sort_desc(lst): return sorted(lst, reverse=True)

动态 Prompt vs 静态 Prompt

在上下文组装中，一个关键设计决策是区分静态 Prompt 和动态 Prompt。这个区分不仅影响信息组织方式，还直接影响缓存命中率和 token 成本。

静态 Prompt
System Prompt 中的固定部分
角色定义、核心规则、输出格式要求
每次对话都一样 → 可命中 Prompt Cache
占用：约 200-500 token
动态 Prompt
根据当前上下文动态组装
用户偏好、最近任务经验、工具列表、召回记忆
每次不同 → 无法命中缓存
占用：约 300-1000 token
为什么要分静态和动态？就像餐厅的菜单——招牌菜（静态）印在菜单上不会变，今日特供（动态）每天根据食材更新。把不变的部分前置，LLM 服务端就能缓存已处理的 prompt 前缀，命中缓存时 token 成本降低 90%。

上下文动态组装流程

Agent 每次调用 LLM 前，都需要动态组装上下文——这不是简单的拼接，而是有严格的顺序和优先级：

🎯 组装优化策略
① 静态部分前置

不变的内容放最前面
→ 命中 Prompt Cache
→ token 成本降低 90%

② 动态部分后置

变化的内容放后面
→ 不影响缓存前缀命中
→ 只增加尾部 token 成本

③ 记忆插在对话历史之前

召回记忆放在对话历史前
→ 避免干扰最新对话
→ 保证当前输入最高权重

动态上下文组装器代码示例 
class ContextAssembler:
    """动态上下文组装器：按优先级组装 Prompt，优化缓存命中"""

    def __init__(self, system_prompt: str, tools: list):
        self.static_prompt = system_prompt  # 静态部分：角色定义+核心规则
        self.tools = tools                   # 工具定义列表
        self.memory_store = VectorStore()    # 长期记忆存储

    def assemble(self, user_input: str, chat_history: list) -> str:
        """组装完整上下文，顺序决定缓存命中率"""
        context_parts = []

        # Step 1: 静态 System Prompt（每次不变，命中缓存）
        context_parts.append(self.static_prompt)

        # Step 2: 注入当前可用工具列表（相对稳定，次优缓存）
        tool_descriptions = self._format_tools(self.tools)
        context_parts.append(f"\n可用工具:\n{tool_descriptions}")

        # Step 3: 召回相关长期记忆（动态，每次不同）
        relevant_memories = self.memory_store.search(user_input, top_k=5)
        memory_text = self._format_memories(relevant_memories)
        context_parts.append(f"\n相关经验:\n{memory_text}")

        # Step 4: 添加当前对话历史（动态增长）
        history_text = self._format_history(chat_history)
        context_parts.append(f"\n对话历史:\n{history_text}")

        # Step 5: 用户当前输入（最后，权重最高）
        context_parts.append(f"\n当前请求: {user_input}")

        return "\n".join(context_parts)

    def _format_tools(self, tools):
        # 固定格式模板 → 保持缓存友好
        return "\n".join(
            f"- {t['name']}: {t['description']}"
            for t in tools
        )

    def _format_memories(self, memories):
        # 固定格式模板 → 记忆内容不同但格式相同
        return "\n".join(
            f"[{m['timestamp']}] {m['content']}"
            for m in memories
        )

    def _format_history(self, history, max_rounds=10):
        # 滑动窗口：只保留最近10轮
        recent = history[-max_rounds:]
        return "\n".join(
            f"[{h['role']}] {h['content']}"
            for h in recent
        )
面试锦囊：动态 Prompt vs 静态 Prompt 的核心区别——静态是"固定不变的指令骨架"，动态是"根据上下文动态填充的血肉"。两者分开组装，前置静态命中缓存，后置动态保证灵活性。

2.9 Harness Engineering（Harness 工程）

Agent 写好了，但怎么确保它可靠运行？这就是 Harness Engineering 要解决的问题——为 Agent 构建"测试 Harness"，验证其性能和可靠性。

Harness 原意是"马具"——套在马身上的缰绳和鞍具。Harness Engineering 就是为 Agent 套上缰绳：测试它跑得稳不稳、方向对不对、会不会失控。

三者不是替代关系，而是层层叠加：好 Prompt 是基础，好 Context 让 Agent 获得正确输入，好 Harness 让 Agent 经得起考验。

📋 八股总结 — 面试高频考点
Q1: Agent 的四大核心模块是什么？它们之间如何协同？
四大模块：LLM（大脑）、记忆（存储）、规划（推理）、工具（执行）。

协同流程：感知层接收输入 → 记忆层检索相关上下文 → 规划层拆解任务制定计划 → 执行层调用工具 → 观察层分析结果 → 更新记忆 → 判断是否继续循环。这个闭环可以多轮执行，直到任务完成。
Q2: Function Calling 的原理是什么？它和直接让 LLM 输出 JSON 有什么区别？
Function Calling 是 LLM 原生支持的能力：你在请求中传入工具的 JSON Schema 描述，LLM 判断需要调用工具时，会输出结构化的函数调用（包含函数名和参数），而不是自然语言。

区别：直接让 LLM 输出 JSON 依赖 prompt 工程，不稳定（可能输出错误格式、多余文字）。Function Calling 是模型层面训练过的能力，输出格式可靠，支持并行调用多个函数。但本质上都是让 LLM 产出结构化的行动指令。
Q3: Agent 的记忆系统有哪几种？各自的特点和实现方式？
① 短期记忆：即 LLM 上下文窗口，存当前对话历史。容量有限（4K~200K token），速度最快。管理策略包括滑窗截断和摘要压缩。
② 长期记忆：通过向量数据库存储，支持语义检索。实现方式：文本 → Embedding → 存入向量库 → 查询时做相似度搜索。容量理论无限，但检索有延迟。
③ 工作记忆：类似便签纸，存当前任务的中间状态和推理过程。任务结束清除。常用实现：scratchpad、临时变量。
Q4: CoT、ToT、ReAct 三种规划策略的区别？分别适用什么场景？
CoT（Chain of Thought）：线性思维链，"Let's think step by step"。简单高效，适合数学推理、逻辑推导等单路径问题。

ToT（Tree of Thought）：树状思维，在每个节点分支探索多个方案，通过评估函数选最优路径。适合需要搜索和回溯的问题，如24点游戏、创意写作。

ReAct：思考→行动→观察的循环，每步可以调用工具获取新信息。适合需要与外部世界交互的任务，如信息检索、代码执行。
Q5: MCP 协议解决了什么问题？它的核心设计是什么？
解决的问题：工具定义的碎片化。以前 LangChain、AutoGPT、CrewAI 各自定义工具格式，一个工具要适配多个框架。

核心设计：MCP 定义了统一的 Client-Server 架构。Agent 框架作为 MCP Client，工具作为 MCP Server，通过 JSON-RPC 2.0 通信。一个 MCP Server 写好后，任何支持 MCP 的框架都能直接用——就像 USB-C 统一了充电器接口。

核心价值：工具复用、标准化通信、解耦 Agent 和工具实现。
Q6: 八种 Agent 设计模式如何分类？各自的核心思想是什么？
八种模式可按核心驱动力分为三类：

推理驱动型：ReAct（思考→行动→观察循环）、Plan-and-Execute（先规划后执行+重规划器）。核心是优化 Agent 的决策流程。

增强驱动型：Tool-Augmented（只在需要时调用工具）、Memory-Augmented（三层记忆协同治理）、RAG（检索增强生成）。核心是为 Agent 补充能力短板。

质量驱动型：Multi-Agent（多Agent协作委派）、Reflective（执行→反思→修正）、Autonomous Loop（目标驱动持续执行）。核心是提升输出质量或自动化程度。
Q7: Context Engineering 的核心原则是什么？它和 Prompt Engineering 有什么本质区别？
Context Engineering 核心原则：
① Just-in-Time：只在需要的时刻注入需要的信息，而非一次性塞满
② 防止 Context Rot：主动管理上下文质量，避免注意力稀释
③ 系统性管理：不只优化单条 Prompt，而是管理整个信息环境

与 Prompt Engineering 的区别：Prompt Engineering 关注"怎么写指令"（静态文本优化），Context Engineering 关注"给LLM什么信息环境"（动态信息管理）。前者是单点优化，后者是系统工程。
Q8: Harness Engineering 在 Agent 开发中的作用是什么？它与前两层工程的关系？
作用：为 Agent 构建测试与可靠性验证体系，包括性能基准测试（量化成功率、token效率）和基础设施保障（超时/重试/沙箱/输出校验）。确保 Agent 在真实环境中稳定运行。

与前两层的关系：三者层层叠加而非替代。Prompt Engineering 优化指令质量（基础），Context Engineering 管理信息环境（增强），Harness Engineering 验证运行可靠性（保障）。好的 Agent 需要三层都到位。
Q9: 动态 Prompt 和静态 Prompt 的主要区别是什么？为什么要分开组装？
主要区别：静态 Prompt 是 System Prompt 中固定不变的部分（角色定义、核心规则、输出格式），每次对话都一样；动态 Prompt 是根据当前上下文动态组装的部分（用户偏好、工具列表、召回记忆），每次不同。

分开组装的原因：① 静态前置可命中 Prompt Cache，token 成本降低 90%；② 动态后置不影响缓存前缀命中；③ 两者分开使信息组织更清晰，便于独立维护和优化。
Q10: 上下文窗口中各部分的 token 占用比例如何？哪部分占用最多？
占用比例：System Prompt（静态）5-10%（约 500-2000 token），工具定义（动态）5-15%（约 500-3000 token），召回记忆 5-10%（约 500-2000 token），对话历史 30-60%（随对话轮数增长），用户当前输入+输出预留 20-30%。

占用最多的部分：对话历史，因为它随对话轮数持续增长，是上下文压力的主要来源，也是 Context Rot 的核心诱因。
Q11: Prompt Caching 的原理是什么？如何通过动静分离优化缓存命中率？
原理：LLM 服务端缓存已处理的 prompt 前缀。当新请求的前缀与缓存匹配时，跳过重复计算，直接使用缓存结果。Anthropic 的 Prompt Caching 命中时 token 成本降低 90%，OpenAI 自动缓存相同前缀。

动静分离优化：将不变的静态部分（System Prompt + 工具定义）放在 prompt 最前面，它们构成稳定的缓存前缀；动态部分（记忆召回 + 对话历史 + 用户输入）放在后面，即使变化也不影响前缀缓存命中。策略包括：① 固定 System Prompt 不频繁改动；② 工具定义保持稳定，不在运行时动态增减；③ 记忆召回使用固定格式模板。
Q12: 如果上下文窗口 128K token，系统提示词 2000，工具定义 3000，如何分配剩余 token？
计算：128K - 2000 - 3000 = 123K 剩余。

分配方案：
① 对话历史：约 60-80K（50-65%）——随对话轮数增长，是主要消耗项，建议设置上限 80K
② 记忆召回：约 5-15K（4-12%）——按需检索，top-5~10 片段，每片段约 500-1500 token
③ 输出预留：约 20-30K（16-24%）——确保 LLM 有足够空间生成完整回答

动态调整原则：对话早期，历史少→可多分配记忆和输出；对话后期，历史增长→压缩旧对话摘要，腾出空间给输出。始终为输出预留 20K+，避免 LLM 因空间不足而截断回答。

2.10 上下文窗口 Token 分配与缓存命中

Prompt Caching：缓存命中的秘密

token 预算分配只是第一步，Prompt Caching 是进一步降低成本的利器：

🧊 Prompt Caching 原理

核心机制：LLM 服务端缓存已处理的 prompt 前缀。当新请求的前缀与缓存匹配时，跳过重复计算，直接使用缓存结果。

Anthropic Prompt Caching

开发者标记缓存边界
命中时 token 成本降低 90%
缓存 TTL：5 分钟（滚动续期）

OpenAI Prompt Caching

自动缓存相同前缀
命中时 token 成本降低 50%
无需开发者标记，自动优化

类比：Prompt Caching 就像浏览器缓存——第一次访问网页要下载所有资源，第二次访问相同网页时，静态资源（CSS/JS/图片）直接从缓存加载，只重新请求动态内容。把 prompt 的"CSS"（静态部分）前置，"动态API数据"（变化部分）后置，缓存就能命中。

策略 做法 效果
固定 System Prompt 不频繁改动角色定义和核心规则 保证最稳定的缓存前缀

Token 预算管理器代码示例 
class TokenBudgetManager:
    """Token 预算管理器：根据上下文窗口大小动态分配 token"""

    def __init__(self, window_size: int = 128000):
        self.window_size = window_size

    def allocate(self, system_tokens: int, tool_tokens: int,
                 memory_tokens: int) -> dict:
        """分配 token 预算，返回各部分上限"""
        fixed = system_tokens + tool_tokens
        remaining = self.window_size - fixed

        # 输出预留：至少 20%，确保 LLM 完整回答
        output_reserve = max(int(remaining * 0.25), 20000)

        # 可用于对话历史和记忆的总额
        available = remaining - output_reserve

        # 对话历史上限：可用额的 80%
        history_limit = int(available * 0.80)

        # 记忆召回上限：可用额的 20%
        memory_limit = min(memory_tokens, int(available * 0.20))

        return {
            "system_prompt": system_tokens,
            "tool_definitions": tool_tokens,
            "memory_recall": memory_limit,
            "chat_history": history_limit,
            "output_reserve": output_reserve,
            "total": self.window_size
        }

    def should_compress(self, current_history_tokens: int,
                        history_limit: int) -> bool:
        """判断是否需要压缩对话历史"""
        return current_history_tokens > history_limit * 0.85
        # 超过85%就启动压缩，留15%缓冲区

# 使用示例
manager = TokenBudgetManager(window_size=128000)
budget = manager.allocate(
    system_tokens=2000,    # System Prompt 约2000 token
    tool_tokens=3000,      # 工具定义约3000 token
    memory_tokens=5000     # 记忆召回上限5000 token
)
# 输出:
# system_prompt:  2000   (1.6%)
# tool_definitions: 3000 (2.3%)
# memory_recall:  4750   (3.7%)
# chat_history:   76250  (59.6%) ← 最大消耗项
# output_reserve: 32000  (25.0%) ← 确保完整输出
# total:          128000
面试锦囊：上下文窗口 token 分配的核心原则——"固定部分最小化，对话历史设上限，输出永远留余地"。宁可压缩历史，不可压缩输出空间，否则 LLM 会截断回答丢失关键信息。
