第11章 Google ADK

Google 的渐进式 Agent 开发框架

11.1 什么是 Google ADK？

ADK（Agent Development Kit）是 Google 推出的 Agent 开发工具包。它的核心理念是渐进式披露（Progressive Disclosure）——从最简单的一行代码到复杂的多 Agent 系统，按需增加复杂度。

简单的事应该简单，复杂的事应该可能。ADK 让你从一行代码开始 Agent 开发，需要时再逐步加入工具、记忆、多 Agent 协作。

11.2 ADK 核心架构

🏗️ ADK 的五大核心组件
1. Agent

核心执行单元。封装了 LLM、指令（instruction）、工具（tools）和子 Agent（sub_agents）。一个 Agent 可以嵌套子 Agent，形成层级结构。

2. Skill

技能工厂。将工具+Prompt+流程封装成可复用的能力包。Skill 可被任何 Agent 引用，实现能力共享。

3. Session & State

会话管理。Session 跟踪对话上下文，State 存储持久状态。支持跨会话记忆和状态恢复。

4. Runner

运行器。管理 Agent 的执行生命周期，包括事件流（Event Stream）、异步 I/O 和错误处理。

5. Artifact

产物系统。Agent 生成的文件、图片、代码等结构化产物。支持多 Agent 之间的产物传递和版本管理。

11.4 ADK 的特色功能

🔄 异步 I/O 原生支持

ADK 从底层就是异步设计。Agent 调用、工具执行、事件流都是 async/await，适合高并发场景。

📦 多模态支持

原生支持文本、图片、音频、视频输入输出。结合 Gemini 的多模态能力，轻松处理混合内容。

🔌 MCP 集成

ADK 原生支持 MCP 协议，可以直接使用任何 MCP Server 提供的工具。

🏗️ A2A 协议

支持 Agent-to-Agent 通信协议，不同框架的 Agent 之间可以互相对话协作。

11.5 OpenAI Agents SDK 对比

Google ADK 和 OpenAI Agents SDK 是 2025 年先后推出的两大 Agent 开发框架。它们的设计哲学不同，但都瞄准了同一个目标：让 Agent 开发更简单。

ADK 重"渐进"——从一行代码到复杂系统，平滑演进；OpenAI SDK 重"安全"——Guardrails 护航，Handoff 交接，每一步都有护栏。

核心概念对比

🟢 Google ADK

  • Agent 定义：Agent(name, model, instruction, tools, sub_agents)
  • 多 Agent 协作：sub_agents 嵌套，父 Agent 直接委派
  • 交接机制：隐式委派——父 Agent 根据指令决定交给哪个子 Agent
  • 安全机制：依赖 LLM 自约束 + instruction 中的规则描述，无显式护栏
  • 运行模式：Runner 管理，Event Stream 流式输出
  • 模型绑定：默认 Gemini，可切换其他模型

🔵 OpenAI Agents SDK

  • Agent 定义：Agent(name, instructions, tools, handoffs)
  • 多 Agent 协作：handoffs 交接，Agent 之间显式转交
  • 交接机制：Handoff 是一等公民，有类型签名和交接协议
  • 安全机制：Guardrails 是一等公民，输入/输出护栏，自动拦截
  • 运行模式：Runner.run()，RunResult 结构化返回
  • 模型绑定：默认 OpenAI 模型，模型无关设计

Handoff vs sub_agents：两种交接哲学

🟢 ADK sub_agents 模型

父 Agent 是编排者，子 Agent 完成任务后结果回传父 Agent。父 Agent 拥有全局视角，可以综合多个子 Agent 的结果。

优点：集中编排、结果整合。缺点：父 Agent 是瓶颈，所有结果必须经过它。

🔵 OpenAI Handoff 模型

Agent A 把对话交接给 Agent B，B 接手后继续对话，用户感知不到切换。交接时 B 获得 A 的对话上下文。

优点：无缝切换、用户体验自然。缺点：无集中编排者，流程难以全局把控。

选择建议：层级化任务需要综合多个子结果 → ADK sub_agents；客服场景需要无缝转接 → OpenAI Handoffs。

11.6 A2A 协议深度解析

A2A（Agent-to-Agent）协议是 Google 推出的跨框架 Agent 互操作标准。它解决了 Agent 世界的"巴别塔"问题——不同框架、不同厂商开发的 Agent 之间无法对话。

MCP 解决的是 Agent 与工具的连接问题，A2A 解决的是 Agent 与 Agent 的连接问题。两者互补，不是竞争。

A2A 的四大核心概念

🤝 A2A 协议关键组件
1. Agent Card

Agent 的"名片"。JSON 格式描述文件，包含 Agent 的身份信息、能力列表、通信端点、认证方式等。其他 Agent 通过 Agent Card 发现和识别这个 Agent。

2. Agent Skill

Agent 的"技能清单"。描述 Agent 能做什么（name、description、input/output schema）。其他 Agent 通过 Skill 描述判断是否需要与这个 Agent 协作。

3. Task

A2A 的核心交互单位。一个 Task 代表一次协作请求，有生命周期：created → working → completed / failed / canceled。Task 支持长时运行和状态追踪。

4. Message & Artifact

Task 中的通信载体。Message 是文本/结构化消息，Artifact 是文件/图片等产物。Agent 之间通过 Message 交换信息，通过 Artifact 传递产物。

Task 生命周期

A2A 与 MCP 的关系

🔧 MCP（Model Context Protocol）

  • 定位：Agent ↔ 工具的连接协议
  • 通信方向：Agent → Tool Server（单向调用）
  • 核心概念：Tool、Resource、Prompt
  • 交互模式：请求-响应，即调即返
  • 典型场景：Agent 调用数据库、搜索引擎、文件系统

🤝 A2A（Agent-to-Agent Protocol）

  • 定位：Agent ↔ Agent 的连接协议
  • 通信方向：Agent ↔ Agent（双向协作）
  • 核心概念：Agent Card、Skill、Task、Message
  • 交互模式：Task 生命周期，可长时运行
  • 典型场景：研究 Agent 请求写作 Agent 协作，客服 Agent 交接给专业 Agent
🔑 MCP + A2A = 完整的 Agent 生态

MCP 让 Agent 连接外部工具（"手"），A2A 让 Agent 连接其他 Agent（"伙伴"）。两者互补：
① Agent 通过 MCP 获取工具能力（数据库、搜索、API）
② Agent 通过 A2A 获取协作能力（与其他 Agent 一起完成复杂任务）
③ 一个 Agent 可以同时使用 MCP 工具和 A2A 协作

类比：MCP = 给人配备工具（锤子、螺丝刀）；A2A = 让人可以找同事帮忙。一个完整的团队既需要工具，也需要协作。

11.7 上下文工程集成

上下文工程（Context Engineering）是 Agent 开发中被低估的关键技能。LLM 的每次决策都依赖当前上下文——上下文太长会丢失关键信息，太短会缺少必要背景。ADK 提供了一套完整的上下文管理机制。

Prompt Engineering 是"说什么"，Context Engineering 是"在什么背景下说"。两者配合才能让 Agent 做出好决策。

Session/State：ADK 的上下文容器

🧠 ADK 上下文管理三层架构
Session（对话层）

管理当前对话的所有消息。每次用户提问和 Agent 回答都记录在 Session 中，形成对话上下文。

作用：短期记忆，对话内可见。

State（状态层）

存储跨对话的持久状态。用户偏好、任务进度、关键决策等存入 State，下次对话可以恢复。

作用：长期记忆，跨对话可见。

Instruction（指令层）

Agent 的角色定义和行为规则。Instruction 始终作为上下文的前缀，指导 Agent 的决策。

作用：身份锚定，永远可见。

上下文压缩策略

当对话上下文超出 LLM 的 token 限制时，ADK 提供四种压缩策略：

✂️ 四种上下文压缩策略
1. 截断（Truncation）

最简单粗暴——超出限制时直接丢弃最早的消息。
优点：实现简单，速度快。
缺点：丢失历史信息，可能导致 Agent "失忆"。

2. 摘要（Summarization）

用 LLM 将旧消息压缩成摘要，保留关键信息。
优点：信息保留率高，Agent 仍能"记得"过去。
缺点：摘要本身消耗 token，且可能有信息失真。

3. 滑动窗口（Sliding Window）

保留最近 N 条消息 + 一条摘要前缀。新消息进来，旧消息退出窗口。
优点：兼顾近期细节和全局概览。
缺点：窗口大小需要调优。

4. 选择性保留（Selective Retention）

根据重要性评分决定保留哪些消息。关键决策、用户偏好高优先级，闲聊低优先级。
优点：最智能，保留最有价值的信息。
缺点：需要重要性评分机制，实现复杂。

上下文压缩策略对比

即时上下文（JIT Context）

JIT Context（Just-In-Time Context）是一种"按需注入"策略——只在 Agent 需要时才注入相关上下文，而不是一开始就把所有信息塞进去。

Python: JIT Context 在 ADK 中的实现 
from google.adk import Agent, Runner, Session
from google.adk.tools import function_tool

# JIT 工具：只在需要时才加载相关信息
@function_tool
def load_user_preferences(category: str) -> str:
    """按类别加载用户偏好，不一次性加载所有"""
    preferences_db = {
        "diet": "用户偏好素食，忌辣",
        "travel": "用户偏好自然风光，不喜欢购物",
        "work": "用户是程序员，偏好安静环境"
    }
    return preferences_db.get(category, "无此类别偏好")

@function_tool
def load_project_context(project_id: str) -> str:
    """按项目ID加载项目背景，避免一次性加载所有项目"""
    # 从数据库或文件系统加载特定项目信息
    return project_db.get(project_id, "项目不存在")

jit_agent = Agent(
    name="jit_assistant",
    model="gemini-2.0-flash",
    instruction="""你是个人助手。
    不要一开始就加载所有用户信息。
    只在需要时才调用 load_user_preferences 或 load_project_context。
    这样可以节省 token 并减少信息干扰。""",
    tools=[load_user_preferences, load_project_context]
)

# 运行：Agent 不会一次性加载所有偏好
runner = Runner(agent=jit_agent)
result = runner.run("推荐一个旅游目的地")
# Agent 只会在需要旅游偏好时才调用 load_user_preferences("travel")
💡 JIT Context 的优势

① 节省 Token：只在需要时加载，避免一次性占用大量上下文空间
② 减少噪声：无关信息不注入，Agent 决策更精准
③ 动态适应：对话方向不确定时，按需获取信息比预加载更灵活
④ 成本可控：每次只加载必要信息，LLM 调用成本更低

11.8 结构化输出

Agent 的决策结果往往是自由文本，但下游系统需要结构化数据。Gemini 的结构化输出能力让 Agent 的决策结果可以直接被程序消费，无需额外的解析步骤。

自由文本是给人看的，结构化输出是给程序用的。Agent 系统中两者都需要——人看解释，程序用数据。

Gemini 的 JSON Schema 约束

Python: ADK 中使用结构化输出 
from google.adk import Agent, Runner
from pydantic import BaseModel
from typing import List, Optional
import json

# ===== 定义输出 Schema =====
class ResearchResult(BaseModel):
    """调研结果的结构化输出"""
    topic: str
    summary: str
    key_findings: List[str]
    confidence: float  # 0.0 ~ 1.0
    sources: List[str]
    next_steps: Optional[List[str]] = None

# ===== Agent 配置结构化输出 =====
research_agent = Agent(
    name="structured_researcher",
    model="gemini-2.0-flash",
    instruction="""你是调研助手。每次调研必须返回结构化结果：
    - topic: 调研主题
    - summary: 一段话总结
    - key_findings: 3-5个关键发现
    - confidence: 置信度(0-1)
    - sources: 信息来源URL列表""",
    output_schema=ResearchResult  # ADK 传入 Schema 约束
)

# ===== 运行并获取结构化结果 =====
runner = Runner(agent=research_agent)
result = runner.run("调研 2025 年 AI Agent 框架发展趋势")

# result 直接是 ResearchResult 对象，无需手动解析
print(f"主题: {result.topic}")
print(f"置信度: {result.confidence}")
for finding in result.key_findings:
    print(f"  发现: {finding}")

结构化输出与 Agent 决策结合

📌 Gemini 结构化输出特点

① 支持 response_mime_type="application/json" 强制 JSON 输出
② 支持 response_schema 用 JSON Schema 约束字段和类型
③ Strict Mode：100% 遵守 Schema，不允许任何偏差
④ 与 ADK 的 output_schema 参数直接集成

⚠️ 使用注意事项

① Strict Mode 下 LLM 无法"自由发挥"，可能影响创造性任务
② Schema 过于复杂会增加延迟和 token 消耗
③ 必须定义 required 字段，可选字段用 Optional
④ 嵌套层级不宜过深（建议 ≤ 3 层）

11.9 实战：用 ADK 构建多 Agent 系统

本节通过一个完整的多 Agent 系统示例，演示 ADK 的 Runner 异步执行、Event Stream 流式输出、错误处理等核心机制。

场景：AI 研究助手系统

Python: 完整的多 Agent 系统实现 
import asyncio
from google.adk import Agent, Runner, Session
from google.adk.tools import function_tool
from google.adk.errors import AgentError, ToolError
from pydantic import BaseModel
from typing import List

# ===== 定义结构化输出 =====
class AnalysisResult(BaseModel):
    topic: str
    trend: str
    data_points: List[str]
    confidence: float

# ===== 工具定义 =====
@function_tool
def search_internet(query: str) -> str:
    """搜索互联网获取最新信息"""
    # 实际项目中对接搜索引擎 API
    return f"搜索结果: {query} 的最新资讯..."

@function_tool
def analyze_data(data_source: str) -> str:
    """对数据源进行统计分析"""
    return f"分析结果: {data_source} 的趋势..."

# ===== 定义子 Agent =====
researcher = Agent(
    name="researcher",
    model="gemini-2.0-flash",
    instruction="你是调研专家。收到主题后，用 search_internet 搜集信息，返回详细调研报告。",
    tools=[search_internet]
)

analyst = Agent(
    name="analyst",
    model="gemini-2.0-flash",
    instruction="你是数据分析师。收到调研结果后，用 analyze_data 深入分析，返回结构化分析结果。",
    tools=[analyze_data],
    output_schema=AnalysisResult  # 结构化输出
)

writer = Agent(
    name="writer",
    model="gemini-2.0-flash",
    instruction="你是技术写手。收到调研和分析结果后，写成一份通俗易懂的报告。"
)

# ===== 主 Agent（协调者） =====
coordinator = Agent(
    name="coordinator",
    model="gemini-2.0-flash",
    instruction="""你是研究项目协调者。
    工作流程：
    1. 先让 researcher 调研
    2. 再让 analyst 分析调研结果
    3. 最后让 writer 写报告
    4. 整合所有结果返回用户""",
    sub_agents=[researcher, analyst, writer]
)

# ===== Runner 异步执行 =====
async def run_research(topic: str):
    """异步运行多 Agent 系统"""
    runner = Runner(agent=coordinator)
    
    try:
        # 方式1: 等待完整结果
        result = await runner.run_async(topic)
        print(f"最终结果: {result}")
        
    except AgentError as e:
        print(f"Agent 执行错误: {e}")
        # 错误处理：尝试简化流程
        try:
            simple_result = await runner.run_async(
                f"简单调研: {topic}"
            )
            print(f"降级结果: {simple_result}")
        except Exception:
            print("降级执行也失败，返回默认结果")
            return "抱歉，系统暂时无法完成调研。"
    
    except ToolError as e:
        print(f"工具调用错误: {e}")
        return f"工具异常，请稍后重试: {e}"
    
    return result

# ===== Event Stream 流式输出 =====
async def run_with_stream(topic: str):
    """流式运行，实时获取中间结果"""
    runner = Runner(agent=coordinator)
    
    print(f"开始研究: {topic}")
    async for event in runner.run_stream_async(topic):
        # Event Stream 中的每一步都会推送
        if event.type == "agent_start":
            print(f"  → {event.agent_name} 开始工作")
        elif event.type == "tool_call":
            print(f"  → 调用工具: {event.tool_name}")
        elif event.type == "agent_result":
            print(f"  → {event.agent_name} 完成")
        elif event.type == "final_result":
            print(f"  → 最终结果就绪")
            return event.content
    
    return "无结果"

# ===== Session 持久化 =====
async def run_with_session(topic: str, session_id: str = None):
    """带会话持久化的运行"""
    session = Session(id=session_id or "new")
    runner = Runner(agent=coordinator, session=session)
    
    result = await runner.run_async(topic)
    
    # Session 自动保存对话历史和状态
    # 下次可以用同一个 session_id 继续对话
    print(f"Session ID: {session.id}")
    print(f"State: {session.state}")  # 查看持久状态
    
    return result

# ===== 主入口 =====
if __name__ == "__main__":
    asyncio.run(run_research("2025年AI Agent框架发展趋势"))
    asyncio.run(run_with_stream("LangGraph vs ADK 对比"))

Runner 执行模式详解

🔄 Runner 的三种执行模式
run()

同步执行，等待完整结果返回。
适合：简单任务、脚本调用、测试。

run_async()

异步执行，返回完整结果。
适合：Web 服务、高并发、需要超时控制。

run_stream_async()

流式执行，实时推送 Event Stream。
适合：长任务、需要中间反馈、UI 实时更新。

错误处理策略

🔴 常见错误类型

① AgentError：Agent 执行异常（LLM 返回错误、子 Agent 失败）
② ToolError：工具调用异常（API 超时、网络错误、参数错误）
③ TimeoutError：整体执行超时（Runner 有 timeout 配置）
④ SchemaError：结构化输出不满足 Schema（Gemini 返回格式错误）

🟢 处理策略

① 降级执行：多 Agent 失败时，简化流程用单 Agent 完成
② 重试机制：ToolError 可重试（设置 max_retries）
③ 超时兜底：Runner 设置 timeout，超时后返回缓存结果
④ Schema 松弛：Strict Mode 失败时切换到宽松模式

📋 八股总结 — 面试高频考点
Q1: Google ADK 的核心理念"渐进式披露"是什么意思？
渐进式披露 = 从简到繁按需增加复杂度。Level 1 一行代码创建纯对话 Agent，Level 2 加工具，Level 3 加记忆，Level 4 加子 Agent，Level 5 加自定义流程。

简单的事应该简单，复杂的事应该可能。开发者不需要一开始就理解所有概念，按需逐步引入。
Q2: ADK 的 Agent 层级嵌套机制是什么？
一个 Agent 可以包含子 Agent（sub_agents），形成树状层级。Root Agent 是编排者，将任务委派给专业子 Agent。

例如：Root Agent → Researcher（调研）+ Writer（写作）+ Reviewer（审核）。每个子 Agent 可以有自己的工具和指令。这种嵌套让多 Agent 协作变得自然——不需要额外的编排框架，Agent 本身就是编排者。
Q3: ADK 的 Skill 工厂解决了什么问题？
Skill 工厂将"工具+Prompt+流程"封装成可复用的能力包。

解决的问题：工具复用。传统方式下，每个 Agent 要重新定义工具和 Prompt。Skill 让一个能力包被多个 Agent 引用，实现"定义一次，到处使用"。

类比：Skill = 函数库，Agent = 程序。函数库可以被多个程序调用。
Q4: ADK 和 LangGraph 在多 Agent 协作上有什么区别？
ADK：通过 Agent 嵌套实现多 Agent 协作。父 Agent 直接包含子 Agent，形成层级结构。无需定义图结构，更自然但灵活性略低。
LangGraph：通过图结构定义 Agent 间的关系。节点=Agent/步骤，边=跳转。更灵活、更可控，但需要显式定义。

ADK 更适合层级化任务委派，LangGraph 更适合复杂流程控制。
Q5: ADK 的异步 I/O 设计有什么优势？
ADK 从底层就是 async/await 设计：
① 高并发：多个 Agent 可以并行执行，不阻塞
② 流式响应：通过 Event Stream 实时推送中间结果
③ 资源效率：等待 LLM 响应时不占用线程，可以处理更多请求
④ 超时控制：异步设计让超时和取消操作更自然
Q6: A2A 协议解决了什么核心问题？它的关键组件有哪些？
A2A 解决的核心问题是跨框架 Agent 互操作——不同厂商、不同框架开发的 Agent 之间无法对话协作。

关键组件：
① Agent Card：Agent 的身份名片，包含名称、能力、端点、认证方式
② Agent Skill：Agent 的技能清单，描述能做什么（name、description、input/output schema）
③ Task：协作的基本单位，有生命周期（created→working→completed/failed/canceled）
④ Message & Artifact：Task 中的通信载体，Message 传递信息，Artifact 传递产物

A2A 与 MCP 互补：MCP 连接 Agent 和工具，A2A 连接 Agent 和 Agent。
Q7: 上下文工程在 ADK 中是如何实现的？四种压缩策略的取舍？
ADK 通过三层架构实现上下文管理：
① Session（对话层）：短期记忆，对话内可见
② State（状态层）：长期记忆，跨对话可见
③ Instruction（指令层）：身份锚定，永远可见

四种压缩策略的取舍：
截断：最简单但信息丢失最多，适合对话不重要时
摘要：信息保留率高但有失真风险和额外 token 成本，适合需要全局概览时
滑动窗口：兼顾近期细节和全局概览，适合大多数场景
选择性保留：最智能但实现复杂，适合高价值对话

实践中推荐滑动窗口 + JIT Context的组合：窗口保留近期对话，JIT 按需加载外部信息。
Q8: Google ADK 与 OpenAI Agents SDK 的核心差异是什么？如何选择？
核心差异在协作模型和安全机制：

协作模型：
ADK 用 sub_agents 嵌套——父 Agent 是编排者，子 Agent 结果回传父 Agent，集中编排但父 Agent 是瓶颈
OpenAI SDK 用 Handoffs 交接——Agent 之间显式转交对话，无缝切换但无集中编排者

安全机制：
ADK 依赖 LLM 自约束 + instruction 规则描述，无显式护栏
OpenAI SDK 的 Guardrails 是一等公民，有 input/output_guardrail，自动拦截违规内容

选择建议：
层级化任务需要综合多个子结果 → ADK
客服场景需要无缝转接 + 安全护栏 → OpenAI SDK
跨框架 Agent 互操作需求 → ADK（A2A 协议原生支持）
