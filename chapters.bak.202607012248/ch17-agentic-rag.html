第17章 Agentic RAG

当 RAG 有了自主判断能力

17.1 传统 RAG 的局限

第6章学了传统 RAG：用户提问 → 检索文档 → LLM 生成回答。流程是固定的，但有很多问题：

传统 RAG 的三大痛点：
  1. 查不准：用户问"对比A和B"，检索器可能只返回A的文档，漏了B
  2. 查不够：复杂问题需要多步检索，传统RAG只检索一次
  3. 查了不用：检索到的文档不相关，但LLM还是硬用它来生成回答

17.2 什么是 Agentic RAG？

Agentic RAG = RAG + Agent 自主决策。把 LLM 变成有判断力的 Agent：

传统 RAG
用户提问
↓ 固定流程
检索（一次）
↓
生成回答
流程固定，无判断
Agentic RAG
用户提问
↓ Agent 决策
要不要检索？
检索什么？够不够？
要不要再查一次？
自主判断，多步检索

17.3 Agentic RAG 的四个核心能力

1. 自主检索决策

不是所有问题都需要检索。"1+1=?"不需要查文档。Agent 判断是否需要检索，避免无意义查询。

2. 查询改写

用户问题可能模糊或不适合检索。Agent 改写为更好的检索查询。如"他怎么样"→"张三的工作表现评价"。

3. 多步检索

复杂问题需要多次检索。如"对比A和B"：先查A，再查B，最后综合比较。Agent 自主决定检索策略。

4. 结果评估

检索到的文档相关吗？够不够？Agent 评估检索结果质量，不够就重新检索，而不是硬凑答案。

Agentic RAG 代码示例

Python: Agentic RAG 实现 
from langgraph.graph import StateGraph, END
from typing import TypedDict

class RAGState(TypedDict):
    question: str
    search_query: str
    documents: list
    answer: str
    attempts: int

def decide_search(state: RAGState):
    """判断是否需要检索"""
    response = llm.invoke(f"""
    用户问题：{state['question']}
    
    这个问题需要检索知识库吗？
    - 常识问题（如1+1）→ 不需要
    - 事实/专业问题 → 需要
    回答 "yes" 或 "no"
    """)
    if "no" in response.lower():
        return "direct_answer"
    return "rewrite_query"

def rewrite_query(state: RAGState):
    """改写查询"""
    rewritten = llm.invoke(f"""
    将以下用户问题改写为更适合检索的查询：
    原始问题：{state['question']}
    改写要求：包含关键词、去除歧义
    """)
    return {"search_query": rewritten.content, "attempts": state.get("attempts", 0) + 1}

def retrieve(state: RAGState):
    """检索文档"""
    docs = vector_store.similarity_search(state["search_query"], k=5)
    return {"documents": docs}

def evaluate_docs(state: RAGState):
    """评估检索结果"""
    eval_result = llm.invoke(f"""
    检索到的文档：
    {state['documents']}
    
    这些文档是否足以回答用户问题：{state['question']}？
    回答 "sufficient" 或 "insufficient"
    """)
    if "insufficient" in eval_result.lower() and state["attempts"] < 3:
        return "rewrite_query"  # 重新检索
    return "generate_answer"     # 生成回答

def generate_answer(state: RAGState):
    """生成回答"""
    answer = llm.invoke(f"""
    基于以下文档回答问题。
    文档：{state['documents']}
    问题：{state['question']}
    """)
    return {"answer": answer.content}

# 构建图
workflow = StateGraph(RAGState)
workflow.add_node("rewrite", rewrite_query)
workflow.add_node("retrieve", retrieve)
workflow.add_node("generate", generate_answer)

workflow.set_entry_point("rewrite")
workflow.add_edge("rewrite", "retrieve")
workflow.add_conditional_edges("retrieve", evaluate_docs)
workflow.add_edge("generate", END)

app = workflow.compile()

17.4 Self-RAG 深度解析与代码实战

Self-RAG（Self-Reflective RAG）由 Akari Asai 等人在 2023 年提出，核心思想是让 LLM 在生成过程中自我反思，通过三类特殊 token 进行内省式质量控制：

Self-RAG 的三类反思 Token：
  1. [Retrieve]：判断是否需要检索。生成时插入此 token 表示"我需要外部知识支撑"
  2. [ISREL]：评估检索文档与问题是否相关。相关→Yes，不相关→No
  3. [ISSUP]：评估生成内容是否被检索文档支撑。有依据→Yes，幻觉→No
  4. [ISREF]：评估最终回答是否完整且有用。完整→Yes，不完整→No

Self-RAG 流程：生成 → 反思 → 检索 → 校正

Self-RAG vs 传统 RAG 工作模式对比：
维度 传统 RAG Self-RAG
检索时机 问题输入后固定检索 生成过程中按需检索（[Retrieve]）
文档评估 无评估，检索到就用 [ISREL] 评估相关性，不相关则丢弃
幻觉检测 无检测机制 [ISSUP] 检测生成内容是否被文档支撑
回答完整性 生成即结束 [ISREF] 评估完整性，不完整则校正
成本 低（1次检索+1次生成） 较高（多次反思+可能的校正循环）
准确率 依赖检索质量 显著提升（幻觉率降低 30-50%）

Self-RAG 代码实战（LangGraph 实现）

Python: Self-RAG LangGraph 实现 
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class SelfRAGState(TypedDict):
    question: str
    documents: list
    generation: str
    is_relevant: str      # [ISREL] 结果
    is_supported: str      # [ISSUP] 结果
    is_complete: str       # [ISREF] 结果
    iterations: int

def retrieve_decision(state: SelfRAGState):
    """[Retrieve] 判断：是否需要检索外部知识"""
    question = state["question"]
    # 简单问题不需要检索，直接生成
    response = llm.invoke(f"""
    判断以下问题是否需要检索外部知识库来回答：
    问题：{question}
    
    判断标准：
    - 常识/计算/定义 → "no_retrieve"（不需要检索）
    - 事实/数据/专业领域 → "retrieve"（需要检索）
    
    只回答 "retrieve" 或 "no_retrieve"
    """)
    decision = response.content.strip().lower()
    if "no_retrieve" in decision:
        return "generate_no_retrieve"
    return "retrieve_docs"

def retrieve_docs(state: SelfRAGState):
    """检索文档"""
    docs = vector_store.similarity_search(state["question"], k=4)
    return {"documents": docs, "iterations": state.get("iterations", 0)}

def grade_documents(state: SelfRAGState):
    """[ISREL] 评估文档相关性"""
    filtered_docs = []
    for doc in state["documents"]:
        evaluation = llm.invoke(f"""
        评估以下文档与问题的相关性：
        问题：{state['question']}
        文档内容：{doc.page_content[:500]}
        
        相关 → "relevant"
        不相关 → "irrelevant"
        """)
        if "relevant" in evaluation.content.lower():
            filtered_docs.append(doc)
    return {
        "documents": filtered_docs,
        "is_relevant": "yes" if filtered_docs else "no"
    }

def generate_with_docs(state: SelfRAGState):
    """基于检索文档生成回答"""
    docs_text = "\n".join([d.page_content for d in state["documents"]])
    generation = llm.invoke(f"""
    基于以下文档回答问题。如果文档不足以回答，说明不足之处。
    文档：{docs_text}
    问题：{state['question']}
    """)
    return {"generation": generation.content}

def generate_no_retrieve(state: SelfRAGState):
    """不检索直接生成（适用于常识问题）"""
    generation = llm.invoke(f"直接回答问题：{state['question']}")
    return {"generation": generation.content, "documents": []}

def grade_generation(state: SelfRAGState):
    """[ISSUP] 评估生成内容是否被文档支撑"""
    if not state["documents"]:
        # 无检索模式，跳过支撑性评估
        return "grade_completeness"
    
    evaluation = llm.invoke(f"""
    评估以下回答是否被提供的文档支撑：
    问题：{state['question']}
    回答：{state['generation']}
    文档摘要：{[d.page_content[:200] for d in state['documents']]}
    
    完全被支撑 → "supported"
    部分幻觉 → "not_supported"
    """)
    result = evaluation.content.strip().lower()
    return {"is_supported": result}
    
    # not_supported → 需要校正
    if "not_supported" in result:
        return "correct_generation"
    return "grade_completeness"

def grade_completeness(state: SelfRAGState):
    """[ISREF] 评估回答完整性"""
    evaluation = llm.invoke(f"""
    评估以下回答是否完整且有用地回答了问题：
    问题：{state['question']}
    回答：{state['generation']}
    
    完整有用 → "complete"
    不完整 → "incomplete"
    """)
    result = evaluation.content.strip().lower()
    return {"is_complete": result}

def correct_generation(state: SelfRAGState):
    """校正回答：重新检索或修正生成"""
    iterations = state.get("iterations", 0) + 1
    if iterations >= 3:
        return {"generation": f"抱歉，无法找到足够的可靠信息来回答：{state['question']}"}
    # 重新检索，使用改写后的查询
    rewrite = llm.invoke(f"""
    原问题：{state['question']}
    原回答（有幻觉）：{state['generation']}
    
    改写问题以便更精确地检索支撑文档：
    """)
    docs = vector_store.similarity_search(rewrite.content, k=4)
    return {
        "question": rewrite.content,
        "documents": docs,
        "iterations": iterations
    }

def should_correct(state: SelfRAGState):
    """路由：是否需要校正"""
    if state.get("is_supported") == "not_supported":
        return "correct"
    if state.get("is_complete") == "incomplete":
        return "correct"
    return "finish"

# 构建 Self-RAG 图
workflow = StateGraph(SelfRAGState)
workflow.add_node("retrieve_docs", retrieve_docs)
workflow.add_node("grade_docs", grade_documents)
workflow.add_node("generate", generate_with_docs)
workflow.add_node("generate_no_ret", generate_no_retrieve)
workflow.add_node("grade_gen", grade_generation)
workflow.add_node("grade_complete", grade_completeness)
workflow.add_node("correct", correct_generation)

workflow.set_entry_point("retrieve_decision_node")
workflow.add_conditional_edges("retrieve_decision_node", retrieve_decision)

workflow.add_edge("retrieve_docs", "grade_docs")
workflow.add_edge("grade_docs", "generate")
workflow.add_edge("generate_no_ret", "grade_complete")
workflow.add_edge("generate", "grade_gen")

workflow.add_conditional_edges("grade_gen", lambda s: 
    "correct" if s.get("is_supported") == "not_supported" 
    else "grade_complete"
)
workflow.add_conditional_edges("grade_complete", lambda s: 
    "correct" if s.get("is_complete") == "incomplete" 
    else END
)
workflow.add_edge("correct", "retrieve_docs")

app = workflow.compile()
💡 Self-RAG 实践要点：
  • 反思 Token 模拟：原始 Self-RAG 使用特殊 token 训练模型，实践中可用 LLM prompt 模拟评估逻辑
  • 校正循环上限：设置最大迭代次数（建议 3 次），防止无限循环
  • 无检索模式：常识问题跳过检索，节省成本和延迟
  • 文档过滤：[ISREL] 环节过滤不相关文档，避免噪声干扰生成

17.5 CRAG 与 Adaptive RAG

Self-RAG 从生成端自反思，而 CRAG 和 Adaptive RAG 从检索端和路由端进行优化：

CRAG：检索后纠正（Corrective RAG）

CRAG 的核心逻辑：检索完成后，先评估检索质量，然后根据评估结果纠正检索策略：

Python: CRAG 实现 
from langgraph.graph import StateGraph, END
from typing import TypedDict

class CRAGState(TypedDict):
    question: str
    documents: list
    web_results: list
    grade: str           # "high" / "medium" / "low"
    answer: str

def retrieve(state: CRAGState):
    """初始检索"""
    docs = vector_store.similarity_search(state["question"], k=5)
    return {"documents": docs}

def grade_retrieval(state: CRAGState):
    """评估检索质量"""
    question = state["question"]
    docs = state["documents"]
    
    # 计算相关性分数
    scores = []
    for doc in docs:
        eval_result = llm.invoke(f"""
        评分以下文档与问题的相关性（0-10分）：
        问题：{question}
        文档：{doc.page_content[:400]}
        只输出数字分数。
        """)
        try:
            score = float(eval_result.content.strip())
        except ValueError:
            score = 5  # 默认中等
        scores.append(score)
    
    avg_score = sum(scores) / len(scores) if scores else 0
    
    if avg_score >= 7:
        return {"grade": "high"}
    elif avg_score >= 4:
        return {"grade": "medium"}
    else:
        return {"grade": "low"}

def use_documents(state: CRAGState):
    """高质量：直接使用检索文档"""
    return {}  # 无需额外处理

def supplement_search(state: CRAGState):
    """中等质量：补充 Web 搜索"""
    web_results = web_search_tool.search(state["question"])
    # 合并检索文档和 Web 结果
    all_docs = state["documents"] + [Document(r) for r in web_results[:3]]
    return {"documents": all_docs, "web_results": web_results}

def fallback_generate(state: CRAGState):
    """低质量：放弃检索，用 LLM 自身知识"""
    return {"documents": []}  # 清空文档，纯 LLM 生成

def generate(state: CRAGState):
    """生成最终回答"""
    if state["documents"]:
        docs_text = "\n".join([d.page_content for d in state["documents"]])
        answer = llm.invoke(f"""
        基于以下文档回答问题：
        文档：{docs_text}
        问题：{state['question']}
        """)
    else:
        # 无文档，纯 LLM 生成（需声明不确定性）
        answer = llm.invoke(f"""
        回答问题（注意：未找到相关文档支撑，请说明回答基于自身知识）：
        问题：{state['question']}
        """)
    return {"answer": answer.content}

def route_by_grade(state: CRAGState):
    """根据质量等级路由"""
    grade = state["grade"]
    if grade == "high":
        return "use_documents"
    elif grade == "medium":
        return "supplement"
    else:
        return "fallback"

# 构建 CRAG 图
workflow = StateGraph(CRAGState)
workflow.add_node("retrieve", retrieve)
workflow.add_node("grade", grade_retrieval)
workflow.add_node("use_docs", use_documents)
workflow.add_node("supplement", supplement_search)
workflow.add_node("fallback", fallback_generate)
workflow.add_node("generate", generate)

workflow.set_entry_point("retrieve")
workflow.add_edge("retrieve", "grade")
workflow.add_conditional_edges("grade", route_by_grade)
workflow.add_edge("use_docs", "generate")
workflow.add_edge("supplement", "generate")
workflow.add_edge("fallback", "generate")
workflow.add_edge("generate", END)

crag_app = workflow.compile()

Adaptive RAG：自适应路由

Adaptive RAG 的核心思想：在检索前就判断问题类型，选择最合适的处理策略。通过决策分类器将问题路由到不同分支：

Adaptive RAG 四种路由策略：
  • No Retrieval：常识/计算问题 → 直接生成，不走检索
  • Single-shot Retrieval：简单事实问题 → 单次检索+生成
  • Multi-step Retrieval：复杂推理问题 → 多步检索+综合
  • Web Search：实时信息问题 → Web 搜索补充
Python: Adaptive RAG 决策分类器实现 
from langgraph.graph import StateGraph, END
from typing import TypedDict

class AdaptiveRAGState(TypedDict):
    question: str
    question_type: str    # "common" / "simple" / "complex" / "realtime"
    documents: list
    answer: str

def classify_question(state: AdaptiveRAGState):
    """问题分类器：判断问题类型"""
    question = state["question"]
    classification = llm.invoke(f"""
    分析以下问题，判断最适合的处理策略：
    
    问题：{question}
    
    分类标准：
    - "common"：常识/计算/定义（如"1+1等于几？"、"什么是Python？"）
    - "simple"：简单事实查询（如"Python 3.12 发布日期？"、"GPT-4 参数量？"）
    - "complex"：需要多步推理/对比/分析（如"对比RAG和Finetuning的优劣？"、"Self-RAG如何减少幻觉？"）
    - "realtime"：需要最新实时信息（如"今天新闻？"、"最新股价？"）
    
    只输出分类标签：common / simple / complex / realtime
    """)
    return {"question_type": classification.content.strip().lower()}

def direct_generate(state: AdaptiveRAGState):
    """常识问题：直接生成"""
    answer = llm.invoke(state["question"])
    return {"answer": answer.content}

def single_retrieve(state: AdaptiveRAGState):
    """简单事实：单次检索+生成"""
    docs = vector_store.similarity_search(state["question"], k=3)
    docs_text = "\n".join([d.page_content for d in docs])
    answer = llm.invoke(f"""
    基于以下文档回答问题：
    文档：{docs_text}
    问题：{state['question']}
    """)
    return {"answer": answer.content, "documents": docs}

def multi_step_retrieve(state: AdaptiveRAGState):
    """复杂推理：多步检索+综合"""
    # 第一步：分解问题
    sub_questions = llm.invoke(f"""
    将以下复杂问题分解为2-3个子问题：
    原问题：{state['question']}
    输出格式：子问题1 | 子问题2 | 子问题3
    """)
    
    # 第二步：分别检索每个子问题
    all_docs = []
    for sub_q in sub_questions.content.split("|"):
        sub_q = sub_q.strip()
        docs = vector_store.similarity_search(sub_q, k=3)
        all_docs.extend(docs)
    
    # 第三步：综合所有文档生成回答
    docs_text = "\n".join([d.page_content for d in all_docs])
    answer = llm.invoke(f"""
    基于以下多源检索文档，综合回答原始问题：
    文档：{docs_text}
    原问题：{state['question']}
    """)
    return {"answer": answer.content, "documents": all_docs}

def web_search_generate(state: AdaptiveRAGState):
    """实时信息：Web搜索+生成"""
    web_results = web_search_tool.search(state["question"])
    results_text = "\n".join([r["content"] for r in web_results[:5]])
    answer = llm.invoke(f"""
    基于以下最新Web搜索结果回答问题：
    搜索结果：{results_text}
    问题：{state['question']}
    """)
    return {"answer": answer.content}

def route_by_type(state: AdaptiveRAGState):
    """根据问题类型路由"""
    qtype = state["question_type"]
    routing = {
        "common": "direct",
        "simple": "single_ret",
        "complex": "multi_ret",
        "realtime": "web_search"
    }
    return routing.get(qtype, "single_ret")

# 构建 Adaptive RAG 图
workflow = StateGraph(AdaptiveRAGState)
workflow.add_node("classify", classify_question)
workflow.add_node("direct", direct_generate)
workflow.add_node("single_ret", single_retrieve)
workflow.add_node("multi_ret", multi_step_retrieve)
workflow.add_node("web_search", web_search_generate)

workflow.set_entry_point("classify")
workflow.add_conditional_edges("classify", route_by_type)
workflow.add_edge("direct", END)
workflow.add_edge("single_ret", END)
workflow.add_edge("multi_ret", END)
workflow.add_edge("web_search", END)

adaptive_app = workflow.compile()

三种高级 RAG 模式对比

维度 Self-RAG CRAG Adaptive RAG
优化方向 生成端自反思 检索端纠正 路由端自适应
核心机制 生成→反思→检索→校正循环 检索→评估质量→纠正策略 分类→路由→差异化处理
关键决策点 生成后：有幻觉吗？完整吗？ 检索后：质量高吗？ 检索前：问题是什么类型？
优势 幻觉检测最强 检索质量保障好 成本效率最优
劣势 成本高、延迟长 依赖评估准确性 分类器可能误判
适用场景 医疗/法律等高准确性要求 知识库质量不确定的场景 混合类型问题的通用系统
LLM调用次数 4-6次（含反思循环） 3-5次（含评估+补充） 2-4次（含分类+处理）

17.6 上下文工程与 Agentic RAG

上下文工程（Context Engineering）是 2025 年兴起的概念：不是简单地把所有检索结果塞进 prompt，而是精心裁剪、改写、压缩，只把最相关的信号喂给 LLM。在 Agentic RAG 中，上下文工程贯穿检索前后全流程：

上下文工程的三层策略：
  1. 检索前：Query 改写 → 让检索器拿到最优查询
  2. 检索中：JIT Context → 按需注入上下文，不是一次全塞
  3. 检索后：高信号提取 → 裁剪噪声，只保留与问题强相关的片段

JIT Context：按需注入上下文

JIT Context（Just-In-Time Context）借鉴软件工程 JIT 编译的理念：不到生成阶段需要上下文时不注入。在 Agentic RAG 中：

❌ 传统做法

检索到10个文档 → 全部塞进 prompt → LLM 被噪声淹没 → 生成低质量回答

✅ JIT Context

检索到10个文档 → 评估相关性 → 过滤为3个高信号文档 → 只注入关键片段 → LLM 聚焦生成

Query 改写策略

Query 改写是上下文工程的第一步，直接影响检索质量。三种主流改写策略：

1. HyDE（Hypothetical Document Embedding）

先让 LLM 生成一个假设性答案，再用这个答案作为查询去检索。假设答案和真实文档在语义空间中更接近，检索效果更好。 

# HyDE: 生成假设文档 → 用假设文档检索
def hyde_retrieve(question: str):
    # Step 1: LLM 生成假设答案
    hypothetical_answer = llm.invoke(f"""
    请对以下问题给出一个假设性答案（即使不确定也尝试回答）：
    问题：{question}
    """)
    
    # Step 2: 用假设答案作为查询向量检索
    docs = vector_store.similarity_search(
        hypothetical_answer.content,  # 用假设答案而非原问题
        k=5
    )
    return docs
2. Multi-Query（多查询并行检索）

从不同角度生成多个检索查询，分别检索后合并结果。覆盖不同语义视角，避免单一查询的盲区。 

# Multi-Query: 生成多个视角的查询 → 并行检索 → 合并去重
def multi_query_retrieve(question: str):
    # Step 1: 生成多个查询视角
    queries = llm.invoke(f"""
    从不同角度生成3个检索查询来回答以下问题：
    问题：{question}
    输出格式：query1 | query2 | query3
    """)
    
    # Step 2: 对每个查询分别检索
    all_docs = []
    for q in queries.content.split("|"):
        docs = vector_store.similarity_search(q.strip(), k=3)
        all_docs.extend(docs)
    
    # Step 3: 去重（基于内容相似度）
    unique_docs = deduplicate_by_content(all_docs)
    return unique_docs[:8]  # 限制最终数量
3. Step-back Query（后退抽象查询）

将具体问题抽象化为更广泛的概念问题先检索，获取背景知识后再回答具体问题。先大后小，从宏观到微观。 

# Step-back: 具体问题 → 抽象为更广泛概念 → 先检索背景再回答
def stepback_retrieve(question: str):
    # Step 1: 抽象化问题
    stepback_query = llm.invoke(f"""
    将以下具体问题抽象为一个更广泛的概念性问题：
    具体问题：{question}
    
    示例：
    "Python 3.12 新增了哪些特性？" → "Python 语言版本迭代的设计原则是什么？"
    "Self-RAG 如何减少幻觉？" → "RAG 系统如何控制生成质量？"
    """)
    
    # Step 2: 用抽象问题检索背景知识
    background_docs = vector_store.similarity_search(
        stepback_query.content, k=5
    )
    
    # Step 3: 结合背景知识 + 具体问题生成回答
    context = "\n".join([d.page_content for d in background_docs])
    answer = llm.invoke(f"""
    背景知识：{context}
    具体问题：{question}
    基于背景知识回答具体问题。
    """)
    return answer.content

检索结果裁剪与高信号提取

检索到的文档往往包含大量噪声。高信号提取的目标：只保留与问题直接相关的片段，丢弃无关内容：

Python: 高信号提取实现 
def extract_high_signal(question: str, documents: list, max_tokens: int = 2000):
    """从检索文档中提取高信号片段"""
    signal_fragments = []
    total_tokens = 0
    
    for doc in documents:
        # Step 1: 按段落分割文档
        paragraphs = doc.page_content.split("\n\n")
        
        for para in paragraphs:
            # Step 2: 评估每个段落与问题的相关性
            relevance = llm.invoke(f"""
            评估以下段落与问题的相关性（高/中/低）：
            问题：{question}
            段落：{para[:300]}
            只输出：high / medium / low
            """).content.strip().lower()
            
            # Step 3: 只保留高相关性段落
            if relevance == "high":
                para_tokens = count_tokens(para)
                if total_tokens + para_tokens <= max_tokens:
                    signal_fragments.append(para)
                    total_tokens += para_tokens
    
    # Step 4: 如果高信号不足，补充中相关性段落
    if total_tokens < max_tokens * 0.5:
        for doc in documents:
            for para in doc.page_content.split("\n\n"):
                # 补充中等相关性段落
                if para not in signal_fragments:
                    # ... 同样评估并补充
                    pass
    
    return "\n\n".join(signal_fragments)

# 更高效的方式：使用 embedding 相似度替代 LLM 评估
def fast_signal_extract(question: str, documents: list, threshold: float = 0.7):
    """基于 embedding 相似度快速提取高信号段落"""
    question_embedding = embedding_model.embed_query(question)
    signal_chunks = []
    
    for doc in documents:
        # 将文档切分为小块
        chunks = split_into_chunks(doc.page_content, chunk_size=200)
        for chunk in chunks:
            chunk_embedding = embedding_model.embed_query(chunk)
            similarity = cosine_similarity(question_embedding, chunk_embedding)
            if similarity >= threshold:
                signal_chunks.append((chunk, similarity))
    
    # 按相似度排序，取前 N 个
    signal_chunks.sort(key=lambda x: x[1], reverse=True)
    return [c[0] for c in signal_chunks[:8]]
⚠️ 上下文工程的成本权衡：
  • LLM评估裁剪：质量最高，但每个段落都需要 LLM 调用，成本高
  • Embedding相似度裁剪：成本低、速度快，但语义理解不如 LLM 精细
  • 混合策略：先用 Embedding 快速过滤（阈值 0.6），再用 LLM 精细裁剪剩余候选，平衡成本和质量

17.7 消息压缩与 RAG 长对话

在 RAG 场景下，长对话面临双重压力：对话历史增长 + 检索文档注入，上下文窗口很快溢出。消息压缩与上下文管理是解决之道：

RAG 场景下的 Context 压缩

RAG 长对话的三大压缩策略：
  • 对话历史压缩：将历史问答摘要为关键信息，而非保留完整对话
  • 文档内容压缩：提取文档中的关键论点/数据，而非全量注入
  • 混合压缩：对话摘要 + 文档摘要 + 当前轮完整上下文
Python: RAG Context 压缩实现 
from typing import TypedDict

class CompressedRAGState(TypedDict):
    question: str
    conversation_summary: str   # 历史对话压缩摘要
    current_documents: list     # 当前轮检索文档
    compressed_docs: str        # 压缩后的文档内容
    answer: str

def compress_conversation_history(history: list, max_tokens: int = 500):
    """压缩对话历史：保留关键信息，丢弃细节"""
    if not history:
        return ""
    
    # 将完整对话拼起来
    full_history = "\n".join([
        f"用户：{h['user']}\n助手：{h['assistant'][:200]}" 
        for h in history[-10:]  # 只取最近10轮
    ])
    
    # LLM 摘要压缩
    summary = llm.invoke(f"""
    将以下对话历史压缩为关键信息摘要（不超过{max_tokens} tokens）：
    保留：决策、结论、用户偏好、关键事实
    丢弃：寒暄、重复确认、过程性描述
    
    对话历史：
    {full_history}
    """)
    return summary.content

def compress_documents(question: str, documents: list, budget_tokens: int = 1500):
    """压缩检索文档：提取与问题直接相关的核心内容"""
    if not documents:
        return ""
    
    # Step 1: 对每个文档提取核心论点
    extracted_points = []
    for doc in documents:
        points = llm.invoke(f"""
        从以下文档中提取与问题直接相关的核心论点和数据：
        问题：{question}
        文档：{doc.page_content[:600]}
        
        输出格式：每行一个核心论点/数据，不超过5行
        """)
        extracted_points.append(points.content)
    
    # Step 2: 合并并去重
    merged = "\n".join(extracted_points)
    
    # Step 3: 如果超出预算，进一步压缩
    if count_tokens(merged) > budget_tokens:
        merged = llm.invoke(f"""
        以下信息需要进一步压缩到{budget_tokens} tokens以内：
        保留最关键的论点和数据，删除重复和次要信息。
        
        信息：{merged}
        """).content
    
    return merged

def build_compressed_context(state: CompressedRAGState):
    """构建压缩后的完整上下文"""
    context_parts = []
    
    # 1. 历史对话摘要（如果有）
    if state.get("conversation_summary"):
        context_parts.append(f"[对话背景] {state['conversation_summary']}")
    
    # 2. 压缩后的文档内容
    if state.get("compressed_docs"):
        context_parts.append(f"[参考文档] {state['compressed_docs']}")
    
    # 3. 当前问题（完整保留）
    context_parts.append(f"[当前问题] {state['question']}")
    
    return "\n\n".join(context_parts)

文档分块策略优化

分块（Chunking）策略直接影响检索质量和上下文利用效率：

策略 原理 优势 劣势 适用场景
固定大小分块 按固定 token 数切分（如 512 tokens），带重叠窗口 简单、稳定 可能切断语义 通用场景
语义分块 按语义边界（段落/章节）切分 语义完整性好 块大小不均匀 结构化文档
递归分块 先按大单位（章节），再按小单位（段落）递归切分 层次清晰 实现复杂 长文档
Proposition分块 LLM 提取每个原子命题作为独立块 检索精度最高 成本极高 高精度场景
父子块 小块用于检索，关联的大块用于生成 检索精准+生成完整 存储开销大 Agentic RAG 推荐
Python: 父子块分块策略实现 
from langchain.schema import Document
from typing import List, Tuple

def parent_child_chunking(
    documents: List[Document],
    parent_size: int = 2000,    # 父块大小（用于生成）
    child_size: int = 300,      # 子块大小（用于检索）
    overlap: int = 50           # 子块重叠
) -> List[Tuple[Document, Document]]:
    """父子块分块：小块检索，大块生成"""
    parent_child_pairs = []
    
    for doc in documents:
        # Step 1: 按父块大小切分
        parent_chunks = split_text(
            doc.page_content, 
            chunk_size=parent_size,
            overlap=200
        )
        
        # Step 2: 每个父块内再切子块
        for parent in parent_chunks:
            child_chunks = split_text(
                parent,
                chunk_size=child_size,
                overlap=overlap
            )
            
            for child in child_chunks:
                # 子块用于检索（存入向量库）
                child_doc = Document(
                    page_content=child,
                    metadata={
                        "parent_content": parent,  # 关联父块
                        "source": doc.metadata.get("source", ""),
                        "chunk_type": "child"
                    }
                )
                parent_child_pairs.append((parent, child_doc))
    
    return parent_child_pairs

# 检索时：用子块向量匹配 → 返回关联的父块内容
def retrieve_with_parent(query: str, k: int = 5):
    """检索子块，返回父块完整内容"""
    # 检索子块
    child_results = vector_store.similarity_search(query, k=k)
    
    # 返回关联的父块（更完整的上下文）
    parent_contents = []
    for child in child_results:
        parent = child.metadata.get("parent_content", child.page_content)
        parent_contents.append(parent)
    
    # 去重
    return list(set(parent_contents))[:5]

检索缓存与 Token 预算分配

生产级 RAG 系统需要Token 预算管理：总上下文窗口有限，如何在对话历史、检索文档、系统指令之间合理分配：

组成部分 预算占比 说明
系统指令 5-10% 角色设定、输出格式要求、安全约束
对话历史 15-25% 最近轮次完整保留 + 早期轮次压缩摘要
检索文档 40-60% 高信号提取后的精炼文档内容
当前问答 10-20% 当前用户问题 + 预留生成空间
预留缓冲 5-10% 避免溢出的安全缓冲区
Python: Token 预算分配与检索缓存 
import hashlib
from functools import lru_cache

class TokenBudgetManager:
    """Token 预算分配器：在有限窗口中合理分配各组成部分"""
    
    def __init__(self, total_budget: int = 8000):
        self.total_budget = total_budget
        self.allocations = {
            "system_prompt": int(total_budget * 0.08),   # 8%
            "conversation":  int(total_budget * 0.20),   # 20%
            "documents":     int(total_budget * 0.55),   # 55%
            "current_qa":    int(total_budget * 0.12),   # 12%
            "buffer":        int(total_budget * 0.05),   # 5%
        }
    
    def allocate(self, system_prompt: str, history: list, 
                 documents: list, question: str) -> str:
        """按预算分配构建最终 prompt"""
        parts = {}
        
        # 1. 系统指令（固定，通常不超预算）
        parts["system"] = system_prompt
        
        # 2. 对话历史（压缩到预算内）
        history_budget = self.allocations["conversation"]
        history_text = self._compress_history(history, history_budget)
        parts["history"] = history_text
        
        # 3. 检索文档（高信号提取到预算内）
        doc_budget = self.allocations["documents"]
        doc_text = self._extract_signal(documents, question, doc_budget)
        parts["documents"] = doc_text
        
        # 4. 当前问题
        parts["question"] = question
        
        return self._assemble(parts)
    
    def _compress_history(self, history: list, budget: int) -> str:
        """压缩对话历史到预算内"""
        # 最近2轮完整保留
        recent = history[-2:]
        recent_text = "\n".join([
            f"用户：{h['user']}\n助手：{h['assistant']}" 
            for h in recent
        ])
        recent_tokens = count_tokens(recent_text)
        
        # 如果预算有余，压缩更早的历史
        remaining = budget - recent_tokens
        if remaining > 200 and len(history) > 2:
            older = history[:-2]
            older_summary = compress_conversation_history(
                older, max_tokens=remaining
            )
            return f"[早期对话摘要] {older_summary}\n\n[近期对话] {recent_text}"
        
        return recent_text
    
    def _extract_signal(self, docs: list, question: str, 
                        budget: int) -> str:
        """高信号提取文档内容到预算内"""
        return extract_high_signal(question, docs, max_tokens=budget)

# 检索缓存：相同/相似查询复用检索结果
@lru_cache(maxsize=100)
def cached_retrieve(query_hash: str, k: int = 5):
    """基于查询哈希的检索缓存"""
    # 注意：实际使用时 query_hash = hash(normalize(query))
    docs = vector_store.similarity_search(query_hash, k=k)
    return docs

def normalize_query(query: str) -> str:
    """查询标准化：去除空格、统一大小写、提取关键词"""
    keywords = llm.invoke(f"""
    从以下问题中提取核心关键词（3-5个）：
    问题：{query}
    输出格式：keyword1 keyword2 keyword3
    """).content.strip()
    return keywords.lower()

def retrieve_with_cache(question: str, k: int = 5):
    """带缓存的检索：相似查询复用结果"""
    normalized = normalize_query(question)
    query_hash = hashlib.md5(normalized.encode()).hexdigest()
    
    try:
        # 先查缓存
        return cached_retrieve(query_hash, k)
    except:
        # 缓存未命中，执行检索
        docs = vector_store.similarity_search(question, k=k)
        return docs

    prompt = budget_mgr.allocate(
        system_prompt="你是专业问答助手。基于文档回答，若文档不足请说明。",
        history=[],  # 实际从外部传入
        documents=[Document(page_content=state["compressed_context"])],
        question=state["question"]
    )
    
    answer = llm.invoke(prompt)
    return {"generation": answer.content}

def self_reflect(state: ProductionRAGState):
    """Self-RAG 自反思：[ISSUP] + [ISREF]"""
    if not state.get("documents"):
        # 无检索模式，跳过支撑性评估
        is_supported = "skip"
    else:
        # 评估支撑性
        eval_sup = llm.invoke(f"""
        回答是否被文档支撑？
        问题：{state['question']}
        回答：{state['generation'][:500]}
        文档摘要：{state['compressed_context'][:300]}
        
        "supported" 或 "not_supported"
        """)
        is_supported = eval_sup.content.strip().lower()
    
    # 评估完整性
    eval_ref = llm.invoke(f"""
    回答是否完整有用？
    问题：{state['question']}
    回答：{state['generation'][:500]}
    
    "complete" 或 "incomplete"
    """)
    is_complete = eval_ref.content.strip().lower()
    
    # 计算置信度
    confidence = 0.9 if is_supported == "supported" and is_complete == "complete" \
                 else 0.5 if is_supported == "supported" or is_complete == "complete" \
                 else 0.2
    
    return {
        "is_supported": is_supported,
        "is_complete": is_complete,
        "confidence": confidence
    }

def correct_answer(state: ProductionRAGState):
    """校正循环：改写查询重新检索"""
    iterations = state.get("iterations", 0) + 1
    if iterations >= 3:
        return {"confidence": 0.1}  # 标记为低置信度
    
    # 改写查询
    rewrite = llm.invoke(f"""
    原问题：{state['question']}
    原回答（有问题）：{state['generation'][:300]}
    问题：{state.get('is_supported', 'unknown')} / {state.get('is_complete', 'unknown')}
    
    改写问题以便更精确检索：
    """)
    
    return {
        "search_query": rewrite.content,
        "iterations": iterations
    }

# ========== 路由函数 ==========
def route_by_type(state: ProductionRAGState):
    """路由：根据问题类型选择分支"""
    qtype = state["question_type"]
    if qtype == "common":
        return "direct_generate"
    elif qtype == "realtime":
        return "web_search"
    else:
        return "retrieve"  # factual / complex

def route_by_grade(state: ProductionRAGState):
    """路由：根据检索质量选择分支"""
    grade = state["retrieval_grade"]
    if grade == "high":
        return "compress"
    elif grade == "medium":
        return "supplement"
    else:
        return "fallback"

def route_by_reflection(state: ProductionRAGState):
    """路由：根据自反思结果选择分支"""
    if state.get("iterations", 0) >= 3:
        return "finish"
    
    if state.get("is_supported") == "not_supported" or \
       state.get("is_complete") == "incomplete":
        return "correct"
    return "finish"

# ========== 构建完整图 ==========
workflow = StateGraph(ProductionRAGState)

# 添加所有节点
workflow.add_node("classify", route_classifier)
workflow.add_node("rewrite", rewrite_for_retrieval)
workflow.add_node("retrieve", retrieve_documents)
workflow.add_node("grade", grade_retrieval_quality)
workflow.add_node("supplement", supplement_retrieval)
workflow.add_node("fallback", fallback_to_llm)
workflow.add_node("compress", compress_context)
workflow.add_node("generate", generate_answer)
workflow.add_node("reflect", self_reflect)
workflow.add_node("correct", correct_answer)
workflow.add_node("direct_generate", lambda s: {
    "generation": llm.invoke(s["question"]).content,
    "confidence": 0.6
})

workflow.set_entry_point("classify")

# 路由分类
workflow.add_conditional_edges("classify", route_by_type)
workflow.add_edge("direct_generate", END)

# 检索流程（factual/complex → rewrite → retrieve → grade）
workflow.add_edge("rewrite", "retrieve")
workflow.add_edge("retrieve", "grade")

# 检索质量路由
workflow.add_conditional_edges("grade", route_by_grade)
workflow.add_edge("supplement", "compress")
workflow.add_edge("fallback", END)  # 低质量直接用LLM

# 压缩 → 生成 → 自反思
workflow.add_edge("compress", "generate")
workflow.add_edge("generate", "reflect")

# 自反思路由
workflow.add_conditional_edges("reflect", route_by_reflection)
workflow.add_edge("correct", "rewrite")  # 校正循环

production_app = workflow.compile()

# ========== 运行示例 ==========
result = production_app.invoke({
    "question": "Self-RAG 和 CRAG 在幻觉检测方面有什么区别？",
    "iterations": 0
})

print(f"回答：{result['generation']}")
print(f"置信度：{result['confidence']}")
print(f"迭代次数：{result.get('iterations', 0)}")
💡 生产级 Agentic RAG 关键配置：
  • 校正循环上限：最多 3 次迭代，防止无限循环和成本爆炸
  • 置信度标签：每条回答附带置信度（0-1），低于 0.3 时建议人工复核
  • Token 预算：8K 窗口时，文档占 55%、对话占 20%、系统占 8%
  • 父子块策略：小块（300 token）检索，大块（2000 token）生成
  • 检索缓存：相似查询复用结果，减少重复检索开销
  • 降级兜底：检索完全失败时，用 LLM 自身知识回答并标注不确定性
📋 八股总结 — 面试高频考点
Q1: 传统 RAG 有什么局限？Agentic RAG 怎么解决？
传统 RAG 痛点：①查不准（检索查询就是用户原话）②查不够（只检索一次）③查了不用（检索结果不相关也硬用）。

Agentic RAG 解决：①自主检索决策（不需要检索就不查）②查询改写（优化检索查询）③多步检索（不够就再查）④结果评估（不相关就重新检索或放弃）。

核心区别：传统 RAG 是固定流程，Agentic RAG 有判断力。
Q2: Agentic RAG 的四个核心能力是什么？
① 自主检索决策：判断是否需要检索，常识问题不浪费检索。
② 查询改写：将用户问题改写为更适合检索的查询。
③ 多步检索：复杂问题分多次检索，逐步收集信息。
④ 结果评估：评估检索结果质量，不够就重新检索。
Q3: Self-RAG 和 CRAG 的区别是什么？
Self-RAG：关注生成后的自反思。回答生成后，Agent 自问"有依据吗？有幻觉吗？"发现问题就修正回答。
CRAG：关注检索后的质量纠正。检索后先评估文档质量：好→用；一般→补充检索；差→放弃检索用 LLM 自身知识。

Self-RAG 纠正的是"回答"，CRAG 纠正的是"检索结果"。
Q4: Agentic RAG 的查询改写为什么重要？
用户原始问题通常不适合直接检索：
① 口语化："他最近怎么样" → 需要明确"他"是谁
② 多意图："对比A和B的优缺点" → 需要拆分为两次检索
③ 关键词缺失："那个新的框架" → 需要补充具体名称

查询改写把用户语言转换为最优检索语言，显著提升检索质量。
Q5: 什么时候用 Agentic RAG，什么时候用传统 RAG？
传统 RAG 适合：简单问答、文档明确、用户问题直接。如FAQ机器人、产品说明书查询。延迟低、成本低。
Agentic RAG 适合：复杂问题、需要多步推理、用户问题模糊。如研究分析、技术调研。延迟高、成本高，但质量好。

权衡：Agentic RAG 用更多时间和 Token 换取更好的回答质量。
Q6: Self-RAG vs CRAG：什么时候用哪个？
Self-RAG 适用：幻觉风险高的场景（医疗、法律、金融），需要逐段验证生成内容的准确性。Self-RAG 的 [ISSUP] 反思机制能精确检测哪些段落是幻觉。
CRAG 适用：知识库质量不确定的场景（混合来源、可能有噪声），需要在检索阶段就过滤低质量文档。CRAG 的三档路由（用/补充/放弃）比 Self-RAG 更灵活处理检索质量问题。

组合使用：生产级系统可两者结合——先用 CRAG 保障检索质量，再用 Self-RAG 检测生成幻觉，双重保障。
Q7: 上下文工程在 RAG 中如何应用？
上下文工程贯穿 RAG 全流程三层：
① 检索前（Query改写）：HyDE 生成假设文档再检索、Multi-Query 多视角并行检索、Step-back 抽象化后检索背景知识。
② 检索后（高信号提取）：Embedding 相似度过滤低相关段落、LLM 评估保留高相关片段、Token 预算管理确保关键内容不被截断。
③ 生成时（JIT Context）：不在检索阶段就全量注入，而是按生成阶段需要按需注入裁剪后的上下文。

核心原则：不是塞更多，而是塞更精——噪声越少，生成质量越高。
Q8: Agentic RAG 的成本如何权衡？
成本来源：① LLM 调用次数（分类+改写+评估+反思+生成）② Token 消耗（多轮反思循环）③ 延迟（多步检索+评估）。

优化策略：
① Adaptive 路由：简单问题只 1 次生成，复杂问题才走完整流程——平均成本降低 40-60%
② 检索缓存：相似查询复用检索结果，避免重复检索开销
③ Embedding 代替 LLM 评估：用向量相似度做粗筛，LLM 只做精筛——评估成本降低 70%
④ Token 预算管理：55% 文档 + 20% 历史 + 8% 系统指令，避免溢出浪费
⑤ 校正循环上限：最多 3 次，防止成本爆炸

经验值：生产级 Agentic RAG 单次问答成本约传统 RAG 的 2-3 倍，但准确率提升 30-50%，幻觉率降低 40%+。
